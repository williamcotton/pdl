# PDL v0.30 Plan

Status: Implemented
Target version: 0.30.0
Owner: PDL maintainers
Related spec: [`PDL_SPEC.md`](PDL_SPEC.md)
Predecessor plan: [`V0_29_PLAN.md`](V0_29_PLAN.md)
Cross-repo coordination: `../studio/` and any browser hosts that consume the
`pdl-wasm` or `pdl-editor` npm packages.

## Purpose

PDL v0.30 prepares the first-party browser npm packages for publication to npm.
The current `packages/wasm/` and `editors/monaco/` packages are useful for
source-mode workspace development, but their package manifests expose
TypeScript source files as runtime entrypoints. Published packages should ship
stable build outputs with CommonJS, ESM, and TypeScript declaration entrypoints,
plus the generated browser WASM artifact where applicable.

The target package contract should follow the local `webpipe-js` publication
shape where it fits, using
`/Users/administrator/Projects/wp/webpipe-js/package.json` as the concrete
manifest model: package metadata points at `dist/index.cjs`, `dist/index.mjs`,
and `dist/index.d.ts`; `exports` provides `types`, `import`, and `require`;
`prepack` builds the publishable surface; and the npm tarball is verified before
publication. The implementation may differ where the browser editor package
needs Monaco, worker, CSS, JSON, or WASM asset handling.

This release is package and build plumbing only. It must not change PDL source
syntax, parser behavior, execution semantics, editor-service behavior, or the
browser JSON ABI except where a wrapper change is needed to make the same
existing API publishable.

## Must

- Publish `pdl-wasm` from generated `dist/` entrypoints.

  Status: Implemented.

  Acceptance criteria:

  - `packages/wasm/package.json` publishes `main`, `module`, `types`, and
    `exports["."]` entries that point at generated `dist/` files rather than
    `src/index.ts`.
  - The built package provides both `dist/index.cjs` and `dist/index.mjs`.
  - The built package provides TypeScript declarations, including the ESM
    declaration entrypoint expected by `types` and any CommonJS declaration file
    generated by the build tool.
  - The package keeps `./package.json` exported for tooling.
  - `sideEffects` remains false unless the implementation introduces a real
    side-effectful runtime import.
  - The public runtime APIs stay source-compatible:
    `defaultPdlWasmUrl`, `loadPdlRuntime`, `PdlRuntime.run`, and
    `PdlRuntime.editorService` remain available with the same TypeScript shapes
    as v0.29.

- Build and package the PDL WASM artifact.

  Status: Implemented.

  Acceptance criteria:

  - A package-local build configuration emits CJS, ESM, and declaration files
    into `packages/wasm/dist/`.
  - The publish/prepack path builds or copies `dist/pdl.wasm` so the npm tarball
    contains a package-local default WASM asset.
  - `defaultPdlWasmUrl()` resolves correctly from the ESM build and has a
    documented, tested behavior from the CommonJS build. If the CommonJS build
    cannot infer a useful package-local URL in all bundlers, callers must still
    be able to pass an explicit `wasmUrl` without runtime failure.
  - The build does not check generated `dist/` files or local tarballs into
    source unless the release process explicitly decides to publish tracked
    artifacts.
  - Package lockfiles are updated for any build tooling added by this release.

- Publish `pdl-editor` from generated `dist/` entrypoints.

  Status: Implemented.

  Acceptance criteria:

  - `editors/monaco/package.json` publishes `main`, `module`, `types`, and
    `exports["."]` entries that point at generated `dist/` files rather than
    `src/index.tsx`.
  - The built package provides both `dist/index.cjs` and `dist/index.mjs`.
  - The built package provides TypeScript declarations, including React
    component prop types and Monaco setup/provider types.
  - Package exports keep `./package.json` and the static editor assets exported:
    `./assets/pdl.tmLanguage.json` and
    `./assets/language-configuration.json`.
  - `peerDependencies` remain peers. React, React DOM, Monaco, TextMate,
    Onigasm, and `pdl-wasm` must not be silently bundled into the library
    entrypoint unless a specific browser asset requires copying.
  - CSS side effects remain declared so downstream bundlers do not tree-shake
    required Monaco/editor styles.

- Make browser asset imports publishable.

  Status: Implemented.

  Acceptance criteria:

  - The editor package build handles current browser-only imports for Monaco
    CSS, Monaco workers, TextMate JSON grammars, and Onigasm WASM URL loading
    without relying on unpublished source files.
  - Vite-specific query imports such as `?worker` and `?url` are either
    converted to a package-friendly strategy or intentionally preserved only if
    the published package contract documents the bundler requirement and the
    build outputs prove usable.
  - Hosts can still opt out of package-provided worker setup through the
    existing setup option when they need custom Monaco worker configuration.
  - The package keeps static editor assets in `assets/` and synchronizes them
    from the canonical editor assets before type/build/package validation.

- Prove npm tarball contents despite generated-output ignore rules.

  Status: Implemented.

  Acceptance criteria:

  - The release chooses an explicit npm packaging strategy for ignored generated
    output. Prefer a `files` whitelist plus `prepack` output verification; add
    package-local `.npmignore` exceptions only if `npm pack --dry-run` proves
    the repository `.gitignore` excludes required `dist/` files from the
    tarball.
  - `npm pack --dry-run` for `pdl-wasm` includes at least:
    `dist/index.cjs`, `dist/index.mjs`, declaration files, `dist/pdl.wasm`,
    `README.md`, and `package.json`.
  - `npm pack --dry-run` for `pdl-editor` includes at least:
    `dist/index.cjs`, `dist/index.mjs`, declaration files, `assets/`,
    `README.md`, and `package.json`.
  - Source files may be excluded from published tarballs unless retained
    intentionally for debugging or source maps.

- Keep spec, versions, and package documentation aligned.

  Status: Implemented.

  Acceptance criteria:

  - `PDL_SPEC.md` records the v0.30 package publication contract and release
    table entry before this plan is marked implemented.
  - Workspace/package version stamps that track the release are aligned to
    `0.30.0`, including Cargo metadata, package manifests, package lockfiles,
    VS Code package metadata if required by the repo release policy, and any
    demo manifests that carry the PDL release version.
  - Package READMEs document published-package consumption, local development,
    `wasmUrl` handling, and the validation commands used before publication.

## Should

- Start from the `webpipe-js` manifest/build pattern, especially
  `/Users/administrator/Projects/wp/webpipe-js/package.json`, then adapt for
  browser package constraints.
- Inspect the neighboring `webpipe-js` build files before implementing,
  including `/Users/administrator/Projects/wp/webpipe-js/tsup.config.ts`,
  `/Users/administrator/Projects/wp/webpipe-js/tsconfig.json`, and
  `/Users/administrator/Projects/wp/webpipe-js/package-lock.json`, plus any
  other adjacent files needed to understand the published `dist/` layout.
- Keep the source-mode development path available for local cross-repo work, but
  make the npm package entrypoint be `dist/`.
- Treat `npm pack --dry-run` output as authoritative for ignore-rule decisions.
  Do not assume a `files` whitelist defeats the repository `.gitignore` until
  the tarball contents are inspected.
- Smoke test both package formats where the entrypoint is runtime-only. The
  `pdl-wasm` ESM import and CommonJS require should load without syntax errors.
  The browser-only Monaco editor package should instead be validated through
  package build, `npm pack --dry-run`, and a Vite/browser host import because
  Monaco pulls CSS, worker, and ESM browser modules.

## Non-Goals

- Publishing to npm from an agent session.
- Changing PDL language syntax, diagnostics, execution behavior, file formats,
  or editor-service semantics.
- Replacing the manual pointer/length WASM ABI with generated `wasm-bindgen`
  bindings.
- Bundling Studio, demo UI, route state, execution buttons, or output panels
  into either npm package.
- Checking generated WASM binaries or local `.tgz` package files into source.

## Validation

Required repository checks before this plan can be marked implemented:

```bash
cargo fmt --all --check
cargo clippy --workspace --all-targets
cargo test --workspace
```

Package validation:

```bash
cd packages/wasm
npm run build
npm pack --dry-run

cd ../../editors/monaco
npm run build
npm pack --dry-run
```

Additional validation:

- Inspect both npm dry-run file lists and confirm required `dist/`, type, WASM,
  README, package metadata, and asset files are present.
- Run ESM import and CommonJS require smoke tests against the built `pdl-wasm`
  entrypoints.
- For `pdl-wasm`, verify that an explicit `wasmUrl` still loads a generated
  `pdl.wasm` in a browser host.
- For `pdl-editor`, verify that a Vite browser host can import the built editor
  package and load Monaco, TextMate grammar assets, Onigasm, and editor
  providers without source-path imports.

## Deferred

- Automated npm publication credentials and registry release automation.
- Source maps, package provenance, or npm provenance signing unless required by
  the release process.
- Separate framework packages beyond the existing Monaco/React editor package.
- Node-native runtime support for executing PDL without browser WASM loading.
