# PDL v0.52 Plan

Status: Shipped
Target version: 0.52.0
Owner: PDL maintainers
Related spec: [`PDL_SPEC.md`](PDL_SPEC.md)
Predecessor plan: [`V0_51_PLAN.md`](V0_51_PLAN.md)

## Purpose

PDL v0.52 brings the deferred reactive parameter metadata back into the
language and ships a local live runner for PDL-driven data preparation
workflows.

PDL already supports top-level `param` declarations, `$name` references,
runtime context maps, and WASM/browser context passing from v0.29. What is
missing is the author-facing control metadata and the local host that can turn
those declarations into native UI controls without hand-written React.

The target workflow is:

```pdl
param min_commits = input_range(label: "Min Commits", min: 0, max: 500, default: 50, step: 10)
param active_author = input_select(label: "Author", choicesFrom: author_line_totals.author_name, default: "all")
param include_bots = input_checkbox(label: "Include Bots", default: false)
param title_filter = input_text(label: "Title Filter", default: "")

let authors_with_min_commits =
  author_line_totals
  | filter commits >= $min_commits
  | select author_name
```

Then:

```bash
pdl serve monthly-author-line-changes-social-api.pdl
```

The command starts a local web app, renders controls from the PDL source,
reruns PDL when controls, source files, or loaded local data files change, and
writes normal `save` outputs so Algraf Preview, VS Code, shell watchers, or any
downstream tool can pick up changed CSVs.

## Release Thesis

v0.52 should make reactive PDL useful without requiring a custom Studio page.

The split is:

- PDL owns parameter control declarations, validation, metadata extraction,
  context coercion, source watching, live reruns, diagnostics, and output
  writes.
- The local `pdl serve` app owns generic browser controls and the live
  browser/backend protocol.
- Algraf remains a downstream renderer that reads files generated by PDL.
- Studio may later reuse the same metadata, but Studio-specific panels,
  routing, and bespoke React controls are not part of v0.52.

The release should preserve the existing deterministic PDL model: controls
supply explicit context values, not environment reads, shell execution, hidden
global state, or source-text substitution.

## Control Model

Control initializers are declaration-only forms. They are accepted only as the
right-hand side of top-level `param` declarations. They produce two outputs:

1. a normal runtime default value for `$name`;
2. control metadata for hosts.

Every v0.52 control lowers to one of the existing PDL context value classes:
string, number, boolean, or null. Date, time, datetime, and color controls are
string-valued controls with validation and UI hints; they do not add new PDL
runtime value types.

Existing literal params remain valid and behave as they do today:

```pdl
param min_commits = 50
```

Literal params have no renderable control metadata unless a later release adds
a separate annotation form.

## Must

- Add the standard parameter control suite.

  Status: Implemented.

  The parser, formatter, AST/CST views, semantic analyzer, IR, CLI JSON
  renderers, editor assets, and spec must support these top-level `param`
  control initializers:

  | Initializer | Runtime value | Required fields | Optional fields |
  | --- | --- | --- | --- |
  | `input_text` | string | `label`, `default` | `placeholder` |
  | `input_textarea` | string | `label`, `default` | `placeholder`, `rows` |
  | `input_number` | number | `label`, `default` | `min`, `max`, `step` |
  | `input_range` | number | `label`, `min`, `max`, `default` | `step` |
  | `input_checkbox` | boolean | `label`, `default` | none |
  | `input_select` | string, number, or boolean | `label`, `default`, `choices` or `choicesFrom` | none |
  | `input_radio` | string, number, or boolean | `label`, `default`, `choices` or `choicesFrom` | none |
  | `input_date` | string | `label`, `default` | `min`, `max` |
  | `input_time` | string | `label`, `default` | `min`, `max`, `step` |
  | `input_datetime` | string | `label`, `default` | `min`, `max`, `step` |
  | `input_color` | string | `label`, `default` | none |

  Static `choices` values may be primitive scalar literals matching the
  parameter default class. Dynamic `choicesFrom` values may resolve primitive
  scalar column values and must be coerced against the parameter default class
  before they become selectable context values.

  Acceptance criteria:

  - `$name` evaluates to the control `default` when no external context
    override is supplied.
  - External context overrides behave the same as v0.29 params and are checked
    against the declared default type.
  - Control initializers outside top-level `param` declarations produce a
    targeted diagnostic.
  - Formatting round-trips every control declaration deterministically.
  - All control metadata can be emitted through a single stable JSON schema.

- Add named-argument and control-literal syntax needed by controls.

  Status: Implemented.

  v0.52 must add the minimal syntax required for control declarations:

  - `:` in named arguments.
  - `.` in `choicesFrom` binding-column references.
  - array literals for static `choices`.
  - named arguments inside declaration-only control initializers.

  General array values and general named-argument function calls remain out of
  scope unless the spec explicitly promotes them. The parser should represent
  these forms as control initializer syntax rather than expanding the ordinary
  expression language.

  Acceptance criteria:

  - `label: "..."`, `min: 0`, `max: 60`, `default: 15`, and `step: 5` parse in
    numeric controls.
  - `choices: ["all", "ebike"]` parses in `input_select` and `input_radio`.
  - `choicesFrom: author_line_totals.author_name` parses as a binding-column
    reference, not as a row expression.
  - Missing commas, unknown arguments, duplicate arguments, and wrong literal
    shapes produce stable diagnostics.
  - Normal row expressions do not gain user-visible named-argument semantics in
    v0.52.

- Validate text, number, boolean, date/time, and color controls.

  Status: Implemented.

  Control validation must happen during semantic analysis and must produce
  stable diagnostics.

  Required validation:

  - `input_text` and `input_textarea` defaults and placeholders must be strings.
  - `input_textarea.rows`, when present, must be a positive integer.
  - `input_number` defaults, min, max, and step must be numbers.
  - `input_range` must have numeric `min`, `max`, and `default`.
  - Numeric `min` must be less than or equal to `max`.
  - Numeric defaults must be inside the inclusive range when both bounds are
    known.
  - Numeric `step`, when present, must be positive.
  - `input_checkbox` defaults must be booleans.
  - `input_date` defaults and bounds must be ISO `YYYY-MM-DD` strings.
  - `input_time` defaults and bounds must be `HH:MM` or `HH:MM:SS` strings.
  - `input_datetime` defaults and bounds must be HTML datetime-local compatible
    strings.
  - `input_color` defaults must be `#RRGGBB` strings.

  Acceptance criteria:

  - Invalid control metadata is rejected by `pdl check`.
  - Runtime context overrides must match the declared runtime value class.
  - The local runner clamps nothing silently; invalid external values produce
    diagnostics rather than hidden coercion.
  - Generated control metadata preserves numeric and boolean values as JSON
    numbers and booleans rather than stringifying them.

- Validate static choice controls.

  Status: Implemented.

  `input_select` and `input_radio` with static `choices` must require:

  - `label`;
  - non-empty `choices`;
  - `default`.

  Choice values must be primitive scalar literals. Every static choice must
  match the default value class. `default` must be one of the static choices
  unless the select also uses dynamic `choicesFrom` and the dynamic default
  policy below applies.

  Acceptance criteria:

  - Empty choices are rejected.
  - Duplicate choices are rejected for deterministic UI.
  - Runtime context overrides for the parameter must match the default value
    class.
  - Generated control metadata preserves choice order.

- Add data-derived choices with `choicesFrom`.

  Status: Implemented.

  `input_select` and `input_radio` must support live choices derived from a
  binding column:

  ```pdl
  param active_author =
    input_select(label: "Author", choicesFrom: author_line_totals.author_name, default: "all")
  ```

  `choicesFrom` references a named binding and a column in that binding. The
  local runner evaluates the referenced binding after parsing/analyzing the
  current source and extracts distinct non-null values from the named column.

  Dynamic choices are post-run metadata. The first UI render may show the
  control in a loading or diagnostics state until the binding can be evaluated.
  When source files, input data files, or upstream controls change, choices are
  recomputed and pushed live to the browser.

  Default policy:

  - If `default` is present and not in the dynamic choices, the UI still uses
    it as the current value until the user chooses another value.
  - Authors may include sentinel defaults such as `"all"` even when the source
    column does not contain `"all"`.
  - The generated metadata must mark whether the current value is present in
    the derived choice set.

  Acceptance criteria:

  - `choicesFrom` can reference bindings declared later in the file.
  - Unknown binding names and unknown columns produce diagnostics.
  - Choice extraction is deterministic: first-seen order after binding
    evaluation, with duplicates removed.
  - Null values are omitted from generated choices.
  - A failed dynamic-choice query does not prevent static controls from
    rendering; the UI surfaces diagnostics and keeps the last known valid
    choice set where possible.

- Add `pdl controls <file> --json`.

  Status: Implemented.

  The CLI must expose a non-server metadata command:

  ```bash
  pdl controls dashboard.pdl --json
  ```

  It emits deterministic JSON describing renderable params, including:

  - parameter name;
  - control kind;
  - label;
  - default value;
  - current value when context is supplied;
  - control-specific fields;
  - static choices or dynamic choice source;
  - dynamic choice status when available;
  - source spans for editor/host mapping;
  - diagnostics.

  The command should also accept repeatable context overrides so hosts can ask
  for metadata and dynamic choices under the same values that will drive a run:

  ```bash
  pdl controls dashboard.pdl --json --context min_commits=100
  ```

  Acceptance criteria:

  - JSON field order is deterministic.
  - Plain literal params are included as non-renderable context declarations or
    omitted according to the spec's chosen contract, but the behavior must be
    documented.
  - Consumers can build a UI without parsing PDL source text themselves.
  - The command exits non-zero on fatal parse/analyze errors and still prints
    diagnostics consistently with other CLI commands.

- Add context overrides to `pdl run`.

  Status: Implemented.

  `pdl run` must accept repeatable context overrides:

  ```bash
  pdl run dashboard.pdl --context min_commits=100 --context active_author="Sean O'Dea"
  ```

  Overrides are coerced against declared param/state defaults using the
  existing runtime context rules.

  Acceptance criteria:

  - Unknown context names produce the existing context diagnostic class.
  - Type mismatches produce a targeted diagnostic.
  - Context override parsing is deterministic and shell-friendly.
  - `pdl run` without overrides behaves exactly as before.

- Add `pdl serve <file>`.

  Status: Implemented.

  `pdl serve` starts a local web server and serves a generic controls app
  backed by the native Rust runtime.

  Required behavior:

  - Bind to `127.0.0.1` by default.
  - Print the local URL.
  - Watch the PDL source file.
  - Watch local files loaded by the PDL program when those paths are
    discoverable from `load` stages.
  - Re-parse and re-analyze when source changes.
  - Re-render controls live when params are added, removed, renamed, reordered,
    or changed.
  - Recompute dynamic choices when their source binding or upstream context
    changes.
  - Rerun the PDL program when controls or watched files change.
  - Write normal `save` outputs through the existing execution path.
  - Stream run status, diagnostics, generated controls, current context, and
    output write results to the browser.

  The browser/backend protocol for v0.52 should be small and explicit:

  - `GET /` serves the bundled app.
  - `GET /api/snapshot` returns the latest server snapshot.
  - `GET /events` streams snapshots over Server-Sent Events.
  - `POST /api/context` updates one or more context values.
  - `POST /api/run` requests a manual rerun.

  Acceptance criteria:

  - Adding a new `param ... input_range(...)` in VS Code causes the browser UI
    to show the new control without restarting `pdl serve`.
  - Editing min/max/default/label updates the existing control without
    restarting.
  - Removing a param removes its control and drops stale context values.
  - Changing a control reruns PDL and rewrites affected CSV outputs.
  - Runtime diagnostics appear in the browser without corrupting stdout data
    behavior for normal `pdl run`.
  - Server logs and browser status make it clear whether the latest run is
    pending, running, succeeded, or failed.

## Should

- Keep the live loop fast and debounced.

  Status: Partially implemented. Browser context changes debounce before POST,
  and the SSE stream serializes snapshots per client; explicit stale-run
  cancellation is deferred.

  Source edits, control changes, and file watcher events should debounce reruns
  so normal typing does not launch a run for every keystroke. The server should
  cancel or supersede stale runs when a newer event arrives.

  Acceptance criteria:

  - Rapid slider movement does not create an unbounded run backlog.
  - The UI shows the newest requested context and the newest completed run
    separately when needed.
  - Failed parses during typing do not clear the last successful output files.

- Keep the served app simple and native.

  Status: Implemented.

  The v0.52 app should use standard browser controls and should not require a
  frontend build step. A small bundled HTML/CSS/JavaScript asset is acceptable
  if it keeps packaging simple and lets `pdl serve` work from the installed CLI.

  Expected native mappings:

  - `input_text` -> single-line text input.
  - `input_textarea` -> textarea.
  - `input_number` -> numeric input.
  - `input_range` -> range input plus numeric readout/input.
  - `input_checkbox` -> checkbox.
  - `input_select` -> select.
  - `input_radio` -> radio group.
  - `input_date` -> date input.
  - `input_time` -> time input.
  - `input_datetime` -> datetime-local input.
  - `input_color` -> color input.

- Preserve host integration boundaries.

  Status: Implemented.

  The local app should not become Studio. It should be a small generic host for
  PDL controls and output generation.

  Out of scope for v0.52:

  - Algraf rendering inside `pdl serve`.
  - Story routing or Studio case-study layout.
  - Custom React component authoring.
  - Chart-event-to-state bindings.

- Keep browser/WASM behavior aligned where practical.

  Status: Deferred. The native CLI/server expose the metadata shape; the spec
  explicitly defers a matching WASM ABI to a later browser package release.

  If the control metadata is exposed through the native CLI, the WASM ABI
  should either expose the same metadata shape or the spec should explicitly
  defer it. Hosts must not need a TypeScript reimplementation of the PDL parser
  to discover controls.

- Update editor/static grammar assets and generated language references.

  Status: Partially implemented. Static TextMate assets and the generated
  `PDL_LANG.md` template recognize the new syntax; rename/reference handling
  for `choicesFrom` remains deferred.

  TextMate grammar, formatter behavior, hover/completion, semantic editor
  services, and `PDL_LANG.md` templates should recognize:

  - every `input_*` control initializer;
  - named control arguments;
  - static choices;
  - `choicesFrom` binding-column references.

  Acceptance criteria:

  - Static highlighting in VS Code does not make valid v0.52 control
    declarations look broken.
  - Rename/reference behavior for bindings accounts for `choicesFrom`
    references if feasible in v0.52; otherwise the limitation is documented.
  - Agent-facing language guidance includes the new control syntax so generated
    `.pdl` files do not use unimplemented UI syntax.

- Add a runnable example matching the intended Algraf Preview workflow.

  Status: Implemented.

  Add an example PDL file that writes CSV outputs controlled by multiple
  standard controls. The example should be small enough for CI but
  representative of the workflow:

  - one text or checkbox control;
  - one numeric control;
  - one static select or radio control;
  - one dynamic `choicesFrom` select or radio control;
  - at least one `save` output.

## Could

- Add `--port` and `--host` to `pdl serve`.

  Status: Implemented.

  The default should remain local-only, but users may need to choose a port or
  bind address.

- Add a `--open` flag.

  Status: Deferred.

  Opens the local browser after the server starts.

- Add persisted control state.

  Status: Deferred.

  The server may remember current control values in a local sidecar or browser
  local storage, but v0.52 should not require persistence for correctness.

- Add a JSON event stream for external tools.

  Status: Implemented as the documented `/events` Server-Sent Events stream.

  The documented Server-Sent Events snapshot stream could let VS Code
  extensions or other hosts subscribe to control metadata and run status without
  using the bundled web UI.

## Validation

Required checks before this plan can be marked landed:

```bash
cargo fmt --all --check
cargo clippy --workspace --all-targets
cargo test --workspace
```

Focused validation should cover:

- Parser acceptance and rejection for every `input_*` control initializer,
  named arguments, static choices, and `choicesFrom`.
- Formatter round-trips for multi-line and single-line control declarations.
- Analyzer diagnostics for missing arguments, duplicate arguments, unknown
  arguments, invalid range bounds, invalid defaults, invalid choices, unknown
  dynamic-choice bindings, and unknown dynamic-choice columns.
- Runtime tests proving control defaults become normal context values.
- CLI tests for `pdl controls --json`.
- CLI tests for `pdl run --context`.
- Server tests for source reload, added/removed params, dynamic choice refresh,
  debounced control changes, diagnostics streaming, and output writes.
- Example tests proving saved CSV files change when context changes.
- Regression tests proving existing literal `param` declarations and
  non-reactive PDL programs behave unchanged.

## Deferred

- Chart-event bindings from Algraf marks into PDL `state`.
- Full Studio integration using the v0.52 metadata.
- Custom component registries or user-authored React controls.
- File picker controls and dynamic path-bound `load` sources.
- General named-argument calls in row expressions.
- General array values outside declaration-only control metadata.
- Remote data watching.
- Incremental execution graph invalidation.
- Auth, sharing, deployment, or multi-user serving.
